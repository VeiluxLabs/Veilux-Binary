# VEILUX Internal Security Audit — June 2026

> **Scope.** A full manual review of the VEILUX workspace at tag `v0.7.3`,
> covering consensus, the untrusted-input ingress, signature/identity crypto,
> block validity, the privacy layer (private execution, attestations, slashing),
> economic arithmetic, the network transport, and every Prism command handler.
>
> **Method.** Each security-critical path was read line by line, the trust
> boundary for each input was identified, and adversarial scenarios were
> constructed. Findings that were exploitable were fixed and covered by a
> regression test before this report was finalized.
>
> **Auditor.** Internal (self) review. This does **not** replace an independent
> third-party audit, which remains a prerequisite before any value-bearing
> mainnet — see "Limitations" at the end.

## C-1 (CRITICAL) — Empty-signature votes bypassed verification

**Location.** `consensus/src/lib.rs`, `Aurora::add_vote` (pre-fix).

**Description.** The vote-ingest path verified a vote's Ed25519 signature only
when the signature field was non-empty:

```rust
if let Some(v) = self.validators.get(&vote.voter) {
    if !vote.signature.is_empty() {          // ← guard skips empty sigs
        verify_bytes(&v.public_key, &vote.signing_bytes(), &vote.signature)?;
    }
}
```

Inbound network votes flow straight from `NetMessage::Vote` →
`RoundMachine::on_vote` → `Aurora::add_vote`. An attacker could therefore send
votes attributed to **any** validator with `signature: []`, which were counted
toward the stake-weighted quorum **without any signature check**.

**Impact.** Total loss of BFT safety. Without `--secure` any network peer, and
even with `--secure` any single Byzantine validator, could fabricate a 2/3+
quorum of forged votes and finalize an arbitrary (or conflicting) block —
double-spend / chain takeover. This is the most severe class of consensus bug.

**Root cause.** The node adds its *own* votes to its local `Aurora` instance
before they are signed (signing happens later, at network broadcast). The empty
-signature exemption was a shortcut to let those local self-votes through, but it
also opened the door to forged network votes.

**Fix.** Split the trusted local path from the untrusted network path:

- `Aurora::add_vote` (network) now **requires** a non-empty signature and
  verifies it against the voter's registered key; a missing or invalid signature
  returns `ConsensusError::BadVoteSignature`. A non-validator voter is rejected
  up front.
- `Aurora::add_local_vote` (new) is the only path that accepts an unsigned vote,
  and it only accepts a vote whose `voter` equals the node's own configured
  identity (`self.me`).
- `RoundMachine::cast_own_vote` uses `add_local_vote`; inbound votes in `on_vote`
  use the strict `add_vote`. Votes are signed (`sign_vote`) before broadcast, so
  honest multi-node consensus is unaffected.

**Tests.** `consensus::tests::rejects_unsigned_network_vote`,
`rejects_forged_signature_for_another_validator`, plus the existing
`commits_at_two_thirds`, `voting_across_views_is_not_equivocation`, and
`driver::four_validators_reach_finality` were updated to sign votes with real
validator identities. **Verified live:** a fresh 4-validator network finalizes
in lockstep at quorum 267 under strict verification.

## S-1 (High) — Forged-majority quorum slash (fixed v0.7.3)

Already fixed in the prior session; see `docs/security.md` §2.15. The
`staking.slash_quorum` proof now carries the full `PrivateEnvelope`, the handler
verifies the commitment, derives the real stakeholder set/count from it, requires
every attester to be a genuine stakeholder, and reconstructs each signed message
from the bound commitment. Regression test
`forged_majority_cannot_slash_an_honest_stakeholder`.

## Informational findings (reviewed, not exploitable as-is)

### I-1 — Private inner command is not re-authorized
`Node::apply_private_envelope` decrypts the inner `Command` and executes it
against the node's **private** sub-state without re-checking the inner command's
signature or that its `submitter` is one of the stakeholders. This is acceptable
by design: only the envelope's stakeholders can decrypt and execute it, the
result lives only in their private sub-state (never the global public state), and
divergence between stakeholders is caught by signed attestations + slashing
(§privacy-model). It is documented here so a future change that lets private
execution touch shared state revisits this.

### I-2 — `proposer_for` modulo bias
`ValidatorSet::proposer_for` uses `(height*31 + round) % active.len()`. For small
validator sets the multiplier interacts predictably with set size, so proposer
selection is not perfectly uniform. This is **cosmetic** for liveness/fairness,
not a safety issue (every node computes the same proposer deterministically). A
future improvement could hash `(height,round)` for better distribution.

### I-3 — Session nonce counter uses `wrapping_add`
The encrypted-transport `Session` derives ChaCha20-Poly1305 nonces from a `u64`
counter via `wrapping_add`. A nonce reuse would only occur after 2^64 frames on a
single long-lived connection — not reachable in practice (connections rotate and
re-key on reconnect, with fresh ephemeral X25519 keys). Documented for
completeness; a hard close at a frame ceiling would remove even the theoretical
concern.

### I-4 — Open dev transport
With no `AuthConfig` (`--secure` omitted) the gossip transport is unauthenticated
and unencrypted ("open (dev)" mode). This is an explicit operator choice for local
development; production validators must run `--secure`. The `submit`/`submitPrivate`
ingress RPCs are likewise unauthenticated by design and rely entirely on the
signature/nonce/commitment checks downstream — appropriate for a public submission
endpoint, but operators should rate-limit/firewall it.

## Areas reviewed and found sound

### N-1 — Token arithmetic
All credit paths use `checked_add` (overflow → error); all debit paths are guarded
by an explicit `balance < amount` check before an unchecked subtraction, so no
underflow is reachable. `move_balance`, `credit`, `debit`, `burn_from`,
`collect_fee` are value-conserving. Balances are `u128`.

### N-2 — Fee / reward / slash math
The fee engine (`charge_fee` → `collect_fee`) caps the charge at the payer's
balance, splits burn/reward with bounded bps, and conserves value. The staking
reward accumulator uses `saturating_*` throughout and a MasterChef-style
debt model. Slashing reduces both self-bond and the global reward weight
(`update_lock`), preventing phantom-stake reward drift. Each offence is recorded
under a unique key to prevent double-slashing.

### N-3 — Block validity & root re-execution
`commit_block` re-executes every command, recomputes both `events_root` and
`state_root`, and rejects on any mismatch — so a lying proposer cannot push a
block whose claimed roots differ from a faithful re-execution. Timestamp is bound
to `[parent, now+2h]`; total gas is bounded by `max_block_gas`. The single
untrusted ingress (`ingest_signed`) enforces payload size, reserved-account
namespace, chain-id, signature, trust-on-first-use key binding, monotonic nonce,
and mempool capacity, in that order.

## Limitations of this audit

- **It is a self-review.** The same author wrote and audited the code; an
  independent third-party audit is still required before mainnet. C-1 shows that
  serious bugs can hide in recently-added code — exactly what an external auditor
  exists to catch.
- **Quorum-based privacy slashing assumes an honest stakeholder majority.** If
  ≥ half of a confidential transaction's stakeholders collude, there is no
  canonical truth to arbitrate — a fundamental limit of quorum systems, not a bug.
- **No formal verification / fuzzing-at-scale** was performed beyond the targeted
  fuzz tests (RLP, EVM). Property-based and differential fuzzing are recommended
  next steps.
- **Cryptographic primitives** (Ed25519, X25519, ChaCha20-Poly1305, BLAKE3,
  keccak/secp256k1) are used via vetted crates; their internals were not audited.
